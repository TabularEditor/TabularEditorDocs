## 3.22.0 新增功能

查看我们的[发布博客](https://tabulareditor.com/blog/tabular-editor-3-june-2025-release)，快速了解本次发布中最重要的更新内容。

（仅限企业版）Tabular Editor 3 现已全面支持 OneLake 上的 Direct Lake，以及 SQL 上的 Direct Lake。 这意味着： 这意味着：

- 表导入向导新增了针对 Fabric Warehouse/Lakehouse 的导入选项。 你不再需要知道要连接的 Warehouse/Lakehouse 的 SQL analytics endpoint。 我们会改为显示你 Workspace 中的所有 Warehouse/Lakehouse 列表，方便你选择要使用的那个。
- 在表导入向导的最后一页，你可以选择使用 OneLake 上的 Direct Lake 模式或 SQL 模式，或将表配置为传统的“导入”模式。
- 当模型包含采用 OneLake 上 Direct Lake 模式的表时，你现在也可以添加导入模式的表，以创建混合模式/混合模型。
- 在预览 Direct Lake 表中的数据时，我们不再限制返回的行数（因此你可以像在导入模式下一样，自由滚动浏览表中的所有数据）。 请注意：执行数据预览时，所有列都会被调入 Fabric capacity 中的语义模型。 如果你担心 Fabric capacity 的内存占用，建议改用 DAX 查询。 请注意：执行数据预览时，所有列都会被调入 Fabric capacity 中的语义模型。 如果你担心 Fabric capacity 的内存占用，建议改用 DAX 查询。

> [!NOTE]
> 我们不再将 Direct Lake 模型配置为使用区分大小写的排序规则。 [!NOTE]
> 我们不再将 Direct Lake 模型配置为使用区分大小写的排序规则。 如果你计划在使用区分大小写排序规则的 Fabric Warehouse 上使用 Direct Lake on SQL，则必须在添加任何表或其他对象之前，手动填写语义模型的 **Collation** 属性。

想了解更多信息，可以看看我们的 [Direct Lake 指南文章](xref:direct-lake-guidance)。

## 3.22.0 中的改进

- 将模型保存为单个文件时，**保存文件** 对话框现在会建议使用“model.bim”作为默认文件名。
- 将多项依赖项更新到最新版本，包括将 [AMO/TOM 更新到 19.98.0.3](https://www.nuget.org/packages/Microsoft.AnalysisServices/)。 此更新修复了工作或学校帐户长期存在的身份验证问题，以及反复出现的登录提示。 此更新修复了工作或学校帐户长期存在的身份验证问题，以及反复出现的登录提示。
- 将新建的 Power BI / Fabric 语义模型的默认兼容级别更改为 **1609**，以便通过模型的 [SelectionExpressionBehavior](https://learn.microsoft.com/en-us/dotnet/api/microsoft.analysisservices.tabular.model.selectionexpressionbehavior?view=analysisservices-dotnet) 属性控制计算组中的 [Selection Expressions](https://powerbi.microsoft.com/en-us/blog/deep-dive-into-selection-expressions-for-calculation-groups/) 的默认行为。
- DAX 查询编辑器现已支持 [`MPARAMETER`](https://dax.guide/st/mparameter/) 关键字，参见 [#1467](https://github.com/TabularEditor/TabularEditor3/issues/1467)。
- 我们的 DAX 编辑器现已支持基于单词的自动完成搜索。 我们的 DAX 编辑器现已支持基于单词的自动完成搜索。 换句话说，如果你在编辑器中输入 `sales ytd`，自动完成现在会建议诸如 `[Sales Margin YTD]`、`[Sales Revenue YTD]` 等度量值。
- 在 Power BI Desktop 2025 年六月更新中，[外部工具现在可以对语义模型执行任何写入操作](https://powerbi.microsoft.com/en-us/blog/power-bi-june-2025-feature-summary/#post-30307-_Toc269410729)。 因此，当连接到该版本或更新版本的 Power BI Desktop 中的模型时，我们不再限制 Tabular Editor 中可用的建模操作。 因此，当连接到该版本或更新版本的 Power BI Desktop 中的模型时，我们不再限制 Tabular Editor 中可用的建模操作。

## 3.22.0 中的错误修复

- 在“保存到文件夹”操作中遇到 IOException 时，新增了异常处理（我们有时需要删除文件夹结构中的某些文件夹，但这可能会被版本控制系统施加的文件系统锁阻止）。
- 修复了一个问题：当“新增功能”页面获得焦点时，某些键盘快捷键（默认和自定义）无法使用。
- 当你分配的自定义快捷键已被其他命令占用时，我们现在会显示警告信息，并从该命令中移除该快捷键，以避免歧义。
- 修复了一个问题：在 DAX 窗口函数中使用未限定的列引用时，语义分析器会错误地显示一条错误信息。 参见 [#1460](https://github.com/TabularEditor/TabularEditor3/issues/1460)。 修复了一个可能导致 **数据刷新** 视图偶尔卡死的问题，需要重启应用才能恢复。 参见 [#1461](https://github.com/TabularEditor/TabularEditor3/issues/1461)。
- 修复了一个可能导致 **数据刷新** 视图偶尔卡死的问题，需要重启应用才能恢复。 参见 [#1461](https://github.com/TabularEditor/TabularEditor3/issues/1461)。
- 修复了若干较小的 DAX 编辑器问题，涉及自动完成、自动格式化以及关键字的语法高亮。
- Query Group 名称（即共享 M 表达式和 M 分区的“显示文件夹”）现在可以修改。 此外，如果将包含 Query Group 的对象粘贴到不含该 Query Group 的模型中，系统会自动创建该 Query Group，从而避免在后续保存或部署模型时出现错误信息。 此外，如果将包含 Query Group 的对象粘贴到不含该 Query Group 的模型中，系统会自动创建该 Query Group，从而避免在后续保存或部署模型时出现错误信息。
- 如果将包含 Shared Expression 引用的对象粘贴到不包含该 Shared Expression 的模型中，我们会移除该引用，以避免在将模型保存/部署到 Analysis Services / Power BI 时出现“对象不存在”错误。
- 修复了一个 bug：将模型元数据保存回服务器后，有时无法撤销之前的更改。

**Tabular Editor 3 主要功能概览：**

- 完全可自定义的 IDE，支持多显示器、Hi-DPI 和主题
- 全新强大的 DAX 代码编辑器，支持自动完成、语法检查、代码折叠等诸多功能
- \*工作区模式：允许你将更改保存到磁盘，并同时将模型元数据同步到 Analysis Services
- \*支持无限滚动预览表数据、创建 PivotGrid，或编写 DAX 查询来浏览模型并测试计算逻辑
- \*设置数据刷新计划
- 在 Provider 和 Structured数据源上更新表架构（没错，甚至包括 M 查询！）
- 创建 Data model 图表
- 创建 DAX 脚本，让你在一个文档中编辑多个度量值或其他计算对象
- 录制 C# Script 并保存为宏（之前称为“自定义操作”）
- 集成 VertiPaq分析器
- DAX调试器
- 集成 DAX优化器
- 代码操作，轻松重构 DAX。

\*=仅在连接到 Analysis Services 或 Power BI 实例时可用
