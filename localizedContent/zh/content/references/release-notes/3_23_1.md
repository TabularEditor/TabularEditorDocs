## 3.23.1 的改进

- 我们在 **新建模型** 对话框的下拉列表中新增了兼容级别 1700（SQL Server 2025）。 此外，Power BI 模型的默认兼容级别已更新为 1702（UDF 和日历需要该级别）。 此外，Power BI 模型的默认兼容级别已更新为 1702（UDF 和日历需要该级别）。

## 3.23.1 的错误修复

- 与 3.23.0 相比，这次更新修复了多个与 UDF 和日历相关的问题。 主要涉及我们内置的 DAX 解析器，同时也对自动完成、格式化等功能做了多项调整与稳定性改进。 主要涉及我们内置的 DAX 解析器，同时也对自动完成、格式化等功能做了多项调整与稳定性改进。
- 我们已将 MSAL（Microsoft.Identity.Client）库的版本降级到 [4.72.1](https://www.nuget.org/packages/Microsoft.Identity.Client/4.72.1)，因为该库的较新版本（用于上一版 Tabular Editor 3）与 AS 客户端库配合不佳，[会导致在使用 Microsoft Entra ID 身份验证选项时应用挂起](https://github.com/TabularEditor/TabularEditor3/issues/1485)。
- 我们移除了对计算表格列执行的 **Duplicate** 和 **Cut/Copy/Paste/Delete** 操作，因为这些对象无法在表中手动添加或移除；强行操作会在将更改保存到 Analysis Services / Power BI 时引发问题。 请改为修改计算表格的 DAX。 请改为修改计算表格的 DAX。
- 选择 **Fabric 镜像数据库** 选项时，表导入向导 UI 现在也会显示镜像的 Azure Databricks 目录。
- 修复了一个问题：打开由旧版 Power BI Desktop 创建的模型时，会出现“The given key 'Extended Properties' was not present in the dictionary”错误。
- 修复了一个错误：在 DAX 脚本对象属性中无法使用前向引用。 也就是说。 如果某个度量值的 `FormatString` 属性引用了在脚本后面才定义的度量值，你就不会再收到错误了。 也就是说。 如果某个度量值的 `FormatString` 属性引用了在脚本后面才定义的度量值，你就不会再收到错误了。

## 从 Tabular Editor 2.x 迁移过来？

观看 [此视频](https://youtu.be/O4ATwdzCvWc)，快速了解 Tabular Editor 3 的主要功能。 另外，别忘了查看我们的 [入门指南](https://docs.tabulareditor.com/onboarding/index.html)。 另外，别忘了查看我们的 [入门指南](https://docs.tabulareditor.com/onboarding/index.html)。

**Tabular Editor 3 主要功能概览：**

- 完全可自定义的 IDE，支持多显示器、Hi-DPI 和主题
- 全新强大的 DAX 代码编辑器，提供自动完成、语法检查、代码折叠等更多功能
- \*工作区模式：可将更改保存到磁盘，并同时将模型元数据同步到 Analysis Services
- \*通过无限滚动预览表数据，创建 PivotGrid，或编写 DAX 查询来浏览模型或测试计算逻辑
- \*计划数据刷新
- 在 Provider 数据源和 Structured数据源上更新表架构（是的，甚至适用于 M 查询！）
- 创建 Data model 图表
- 创建 DAX 脚本，以便在单个文档中编辑多个度量值或其他计算对象
- 录制 C# Script 并保存为宏（以前称为“自定义操作”）
- VertiPaq分析器集成
- DAX调试器
- DAX优化器集成
- 通过代码操作轻松重构 DAX。

\*=仅在连接到 Analysis Services 或 Power BI 实例时可用
