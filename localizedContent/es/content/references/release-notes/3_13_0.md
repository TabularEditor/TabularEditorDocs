### Requisitos previos

Tabular Editor 3.3.0 (y versiones posteriores) usa .NET 6. Es posible que tengas que instalar el [Runtime de .NET 6.0 para escritorio](https://dotnet.microsoft.com/en-us/download/dotnet/6.0/runtime) antes de poder iniciar Tabular Editor 3.3.0 (o una versión posterior). A partir de la versión 3.3.1, el instalador de Tabular Editor 3 descargará e instalará estos requisitos previos si no están presentes.

## Novedades en 3.13.0

Consulta nuestra [entrada del blog de la versión](https://blog.tabulareditor.com/2023/12/15/tabular-editor-3-december-2023-release/) para obtener un breve resumen de las actualizaciones más importantes de esta versión.

## Mejoras en 3.13.0

- Las advertencias de variables sin usar ahora solo se muestran cuando se completa el bloque de declaración de variables, y solo se resalta el nombre de la variable. Además, hay una nueva opción con la que puedes desactivar estas advertencias en **Tools > Preferencias > Editor de DAX > Code Assist**.
- El Analizador VertiPaq de TE3 ahora usa la misma configuración predeterminada que en DAX Studio. Ahora también puedes configurar la opción "Read statistics from DirectQuery tables" en **Tools > Preferencias > Analizador VertiPaq**.
- Hemos hecho un pequeño cambio en la forma en que aparece la política de actualización de una tabla en la vista **Properties**, para que se alinee mejor con [TOM](https://learn.microsoft.com/en-us/dotnet/api/microsoft.analysisservices.tabular.table.refreshpolicy?view=analysisservices-dotnet). Este cambio también te permite agregar anotaciones/propiedades extendidas a las políticas de actualización, lo que antes no era posible. Los C# Scripts aún pueden usar las propiedades de la política de actualización directamente en el objeto `Table`, pero te recomendamos actualizar los scripts para acceder a estas propiedades a través del objeto `Table.RefreshPolicy`.
- Al seleccionar una tabla o una columna en el Explorador TOM, ahora resaltamos (en **negrita**) cualquier relación en la que intervenga el objeto; consulta [#865](https://github.com/TabularEditor/TabularEditor3/issues/865). Puedes desactivar esta función en **Tools > Preferencias > Explorador TOM**.
- Ahora mostramos el modo de almacenamiento principal (Import, DirectQuery, Direct Lake, etc.) del modelo, en la barra de título.
- Además de las tablas, ahora puedes ver las relaciones a partir de una columna específica en la vista "Dependencies". Haz clic con el botón derecho en una columna y elige "Show dependencies" y, a continuación, selecciona la opción "Show relationships starting from this column"; consulta [este comentario](https://github.com/TabularEditor/TabularEditor3/issues/865#issuecomment-1490782086).
- Ahora mostramos una advertencia en la vista de **mensajes** cuando los objetos están en un estado sin procesar. Esto se aplica a tablas, columnas y particiones. También soluciona [#1153](https://github.com/TabularEditor/TabularEditor3/issues/1153).
- Ahora mostramos una advertencia en la vista **Mensajes** cuando una columna utiliza otra columna como columna de ordenación y esta tiene `IsAvailableInMDX` establecido en **false** (ya que, de lo contrario, se produciría un error al guardar o implementar el modelo). Consulta [#868](https://github.com/TabularEditor/TabularEditor3/issues/868).
- Hemos actualizado AMO/TOM para usar la versión más reciente [19.72.0](https://www.nuget.org/packages/Microsoft.AnalysisServices.retail.amd64/). Esta biblioteca también incluye una versión actualizada de TMDL, que debería corregir un problema de deserialización cuando una tabla tiene aplicada una política de actualización incremental.
- Al serializar un modelo con TMDL, ahora también generamos un archivo `database.tmdl`, que contiene metadatos a nivel de base de datos (nombre, modo de compatibilidad, nivel de compatibilidad, etc.), lo que debería mejorar la experiencia general con TMDL.

> [!NOTE]
> **TMDL sigue en versión preliminar y no debería usarse en producción, ya que podrías verte afectado por cambios incompatibles entre actualizaciones.**
> Si tienes algún problema al abrir un modelo guardado como TMDL con una versión anterior de Tabular Editor, una posible solución es abrir el modelo desde TMDL con esa misma versión anterior de Tabular Editor, guardarlo como un archivo Model.bim y, a continuación, usar la versión más reciente de Tabular Editor para abrir el archivo Model.bim y guardarlo de nuevo en formato TMDL.

## Correcciones de errores en 3.13.0

- Se corrigió un problema por el que, al deshacer (Ctrl+Z) la eliminación de una política de actualización de una tabla, no se restauraba la configuración de esa política de actualización.
- Se corrigió un problema por el que el autocompletado del Editor de DAX sugería elementos incorrectos para los parámetros de las funciones de ventana, cuando la función de ventana se usaba como argumento de filtro para [`CALCULATE`](https://dax.guide/calculate/) o [`CALCULATETABLE`](https://dax.guide/calculatetable/).
- Se corrigió un error que impedía agregar o quitar niveles a las jerarquías cuando se estaba conectado a una instancia de Power BI Desktop, aunque se trata de una [operación de modelado compatible](https://learn.microsoft.com/en-us/power-bi/transform-model/desktop-external-tools#data-modeling-operations).
- Se corrigió un problema por el que la opción de actualización "Calculate" no aparecía para las tablas normales.
- Se corrigió un problema por el que no era posible evaluar solo la selección actual de una Consulta DAX si había un comentario de línea directamente encima de la primera instrucción `EVALUATE` de la selección; consulta [#912](https://github.com/TabularEditor/TabularEditor3/issues/912).
- Ahora Tabular Editor te permite "Open from folder" cuando la carpeta contiene archivos ".tmdl" (las versiones anteriores de TMDL usaban la extensión ".tmd"). Consulta [#1175](https://github.com/TabularEditor/TabularEditor3/issues/1175).
- La lógica para determinar si una tabla es una "Date Table" era demasiado restrictiva, lo que en algunos casos podía provocar falsos errores de dependencia circular o que el Depurador de DAX mostrara de forma incorrecta los contextos de filtro reemplazados (debido a la eliminación implícita de filtros asociada a las tablas de fechas).

## Vienes de Tabular Editor 2.x?

Mira [este vídeo](https://www.youtube.com/watch?v=pt3DdcjfImY) para hacerte una idea de las nuevas características de Tabular Editor 3. Además, asegúrate de consultar nuestra [guía de incorporación](https://docs.tabulareditor.com/onboarding/index.html).

**Resumen de las principales características de Tabular Editor 3:**

- IDE totalmente personalizable, con compatibilidad con varios monitores, pantallas Hi-DPI y temas
- Nuevo y potente editor de código DAX con autocompletado, comprobación de sintaxis, plegado de código y mucho, mucho más
- \*Modo del área de trabajo, que te permite guardar los cambios en disco y sincronizar los metadatos del modelo con Analysis Services al mismo tiempo
- \*Previsualiza los datos de las tablas con desplazamiento infinito, crea PivotGrids o escribe Consultas DAX para explorar el modelo o probar la lógica de cálculo
- \*Programa actualizaciones de datos
- Actualiza los esquemas de las tablas tanto en Data sources de proveedor como en orígenes de datos estructurados (sí, ¡incluso para consultas M!)
- Crear diagramas del Data model
- Crear scripts DAX que permitan editar varias medidas u otros objetos calculados en un único documento
- Grabar C# Scripts y guardarlos como macros (antes conocidas como "Custom Actions")
- Integración con el Analizador VertiPaq
- Depurador de DAX

\*=Solo mientras se esté conectado a una instancia de Analysis Services o Power BI
